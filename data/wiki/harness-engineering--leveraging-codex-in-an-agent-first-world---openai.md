# Harness Engineering: Leveraging Codex in an Agent-First World

## Summary

**Harness engineering** is an approach to software development where [[AI agents]] (specifically [[Codex]]) handle the majority of code generation and development tasks, while human engineers focus on designing systems, specifying intent, and building feedback loops. This methodology emerged from an OpenAI experiment that built a production software product with zero manually-written lines of code across five months, achieving approximately 10x faster development velocity than traditional approaches.

## Overview

In a groundbreaking internal experiment conducted from August 2025 to February 2026, OpenAI's engineering team developed a complete software product without any human-written code. The project produced approximately one million lines of code—including application logic, tests, [[CI/CD]] configuration, documentation, and internal tooling—entirely generated by [[Codex]] agents.

The core insight driving this approach is philosophical: **humans steer, agents execute**. Rather than writing code directly, engineers shifted their focus to creating the conditions under which [[AI agents]] could work reliably at scale.

## Key Principles

### Redefining the Engineer's Role

Traditional software engineering emphasized individual code authorship and implementation details. Harness engineering reframes the engineer's primary responsibilities as:

- **Systems design**: Creating architectural structures and abstractions
- **Environment scaffolding**: Building the tools and frameworks agents need
- **Feedback loop design**: Establishing mechanisms for validation and iteration
- **Intent specification**: Translating business requirements into agent-legible directives

Rather than asking "how do I implement this feature?", engineers ask "what capability is missing that prevents the agent from implementing this feature?"

### Agent-First Development Workflow

The typical workflow involves:

1. Engineer describes a task via prompt
2. Agent executes the task and opens a pull request
3. Agent reviews its own changes locally
4. Agent requests additional reviews from other agents
5. Agent responds to feedback and iterates
6. Agent merges the change once all reviewers are satisfied

Humans may review pull requests but are not required to. Over time, nearly all review effort migrated to agent-to-agent interactions.

## Critical Success Factors

### Making Applications Legible to Agents

As [[QA]] bottlenecks emerged, the team invested in making application state directly observable to [[Codex]]:

- **UI legibility**: Integrated [[Chrome DevTools Protocol]] to enable agents to interact with the application, capture screenshots, navigate the DOM, and validate UI behavior
- **Observability**: Exposed logs, metrics, and traces to agents via local [[observability]] stacks, allowing agents to query logs with [[LogQL]] and metrics with [[PromQL]]
- **Isolated execution**: Each agent task runs in an isolated [[git worktree]] with its own ephemeral application instance, logs, and metrics

This enabled agents to reproduce bugs, validate fixes, and reason about application behavior directly, sometimes running single tasks for over six hours.

### Repository Knowledge as System of Record

Context management emerged as one of the largest challenges in agent effectiveness. The team discovered that monolithic instruction files failed because:

- Context is scarce; large documents crowd out task and code details
- Over-specification becomes non-specification (everything seems equally important)
- Documentation rots quickly and becomes unreliable
- Monolithic formats resist mechanical verification

**Solution**: Treat the repository itself as the system of record, organized into a structured knowledge base:

```
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
├── DESIGN.md
├── FRONTEND.md
├── PRODUCT_SENSE.md
└── QUALITY_SCORE.md
```

A brief [[AGENTS.md]] file (roughly 100 lines) serves as a table of contents, pointing agents to deeper sources of truth. This enables **progressive disclosure**: agents start with a small, stable entry point and learn where to look next.

**Enforcement**: Custom linters and [[CI]] jobs validate that documentation is:
- Up to date
- Cross-linked
- Correctly structured
- Verified for accuracy

A recurring "doc-gardening" agent scans for stale documentation and opens fix-up pull requests.

### Strict Architectural Constraints

Rather than micromanaging implementations, the team enforced invariants through rigid architectural models:

- Each business domain is divided into fixed layers: **Types → Config → Repo → Service → Runtime → UI**
- Strict validation of dependency directions ensures code only depends "forward" through permitted layers
- Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through a single explicit interface: **Providers**
- All violations are mechanically enforced via custom linters and structural tests

Additional "taste invariants" enforce:
- Structured logging
- Naming conventions for schemas and types
- File size limits
- Platform-specific reliability requirements

These constraints are encoded as custom lints with remediation instructions injected into agent context, making them multipliers: once encoded, they apply everywhere automatically.

### Enforcing Quality Through Continuous Cleanup

Full agent autonomy creates entropy—agents replicate existing patterns, including suboptimal ones. Rather than manual Friday cleanup sessions, the team encoded "golden principles" and automated garbage collection:

**Golden Principles** (opinionated, mechanical rules):
- Prefer shared utility packages over hand-rolled helpers
- Validate data boundaries; don't probe "YOLO-style"
- Maintain consistent patterns across the codebase

**Garbage Collection Process**: Background [[Codex]] tasks run on a regular cadence to:
- Scan for deviations from golden principles
- Update quality grades
- Open targeted refactoring pull requests

Most cleanup PRs can be reviewed in under a minute and auto-merged. This approach treats technical debt as a high-interest loan—better to pay down continuously in small increments than let it compound.

## Operational Changes

### Modified Merge Philosophy

As agent throughput increased, conventional engineering norms became counterproductive:

- **Minimal blocking merge gates**: Pull requests are short-lived
- **Test flake tolerance**: Flakes are addressed with follow-up runs rather than blocking progress indefinitely
- **Rapid iteration**: Corrections are cheap; waiting is expensive

This philosophy works because agent throughput far exceeds human attention capacity, making quick iteration more valuable than perfect first attempts.

## Levels of Autonomy

The system progressively increased agent autonomy as more of the development loop was encoded:

1. **Task execution**: Agents implement features from prompts
2. **Self-review**: Agents review their own changes
3. **Peer review**: Agents request and respond to agent feedback
4. **End-to-end autonomy**: Given a single prompt, agents can:
   - Validate current codebase state
   - Reproduce reported bugs with video documentation
   - Implement fixes
   - Validate fixes by driving the application
   - Open pull requests
   - Respond to feedback
   - Detect and remediate build failures
   - Escalate to humans only when judgment is required
   - Merge changes

**Important caveat**: This level of autonomy depends heavily on specific repository structure and tooling investments and should not be assumed to generalize without similar infrastructure.

## What Gets Generated

"Agent-generated" encompasses:

- Product code and tests
- [[CI/CD]] configuration and release tooling
- Internal developer tools
- Documentation and design history
- Evaluation harnesses
- Review comments and responses
- Repository management scripts
- Production dashboard definitions

## Key Insights

### Agent Legibility as Design Principle

The entire codebase is optimized for [[Codex]]'s legibility first, humans second. This mirrors how teams improve code navigability for new hires, but with agents as the primary audience.

**Principle**: Anything the agent cannot access in-context while running effectively doesn't exist. Knowledge in external systems (Google Docs, chat threads, people's heads) is inaccessible. Only repository-local, versioned artifacts are legible.

This drove decisions about:
- Preferring "boring" technologies with better composability and training data representation
- Reimplementing functionality rather than working around opaque upstream behavior
- Pushing more context into the repository over time

### Discipline Shifts to Scaffolding

Building software with agents still demands discipline, but discipline manifests differently:

- **Traditional approach**: Discipline in code writing and implementation
- **Agent-first approach**: Discipline in tooling, abstractions, and feedback loops

The scaffolding that keeps the codebase coherent becomes increasingly important as agent autonomy increases.
