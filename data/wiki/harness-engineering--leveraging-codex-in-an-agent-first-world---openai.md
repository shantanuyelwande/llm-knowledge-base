# Harness Engineering: Leveraging Codex in an Agent-First World

## Summary

**Harness engineering** is an approach to software development that prioritizes designing systems, environments, and feedback loops for [[AI agents]] to execute code generation at scale, rather than humans writing code manually. This methodology was demonstrated through an experimental product built entirely by [[Codex]] agents—approximately one million lines of code across five months with zero manually-written code. The approach fundamentally redefines the engineer's role from code author to system architect, emphasizing repository structure, documentation, tooling, and mechanical enforcement of architectural constraints.

---

## Overview

In a landmark experiment conducted between August 2025 and February 2026, a team at [[OpenAI]] built a complete software product without writing a single line of code by hand. Every component—from application logic and tests to CI configuration, documentation, and internal tooling—was generated by [[Codex]] agents powered by [[GPT-5]].

The key insight driving this work was fundamental: **humans steer, agents execute**. Rather than attempting to make agents write code in the way humans do, the team redesigned the entire engineering workflow around what agents do best: rapidly generating, testing, and iterating on code when given clear specifications, tools, and feedback mechanisms.

### Key Metrics

- **Codebase size**: ~1 million lines of code
- **Development timeline**: 5 months
- **Pull requests merged**: ~1,500
- **Team size**: 3 engineers (later scaling to 7)
- **Average throughput**: 3.5 PRs per engineer per day
- **Estimated time savings**: 10x faster than manual development

## Redefining the Engineer's Role

### From Code Author to System Architect

The absence of manually-written code introduced a fundamentally different type of engineering work. Rather than writing application code, engineers focused on:

- **Systems design**: Creating the scaffolding and structure agents operate within
- **Abstraction building**: Defining reusable components and patterns
- **Feedback loop design**: Establishing mechanisms for agents to validate their work
- **Intent specification**: Translating high-level goals into actionable prompts

### The Agent Workflow

Human engineers interact with [[Codex]] almost entirely through [[prompts]]:

1. Engineer describes a task
2. Agent executes the task and opens a pull request
3. Codex reviews its own changes locally
4. Agent requests additional reviews from other agents
5. Agent responds to feedback and iterates
6. Loop continues until all agent reviewers are satisfied
7. Human review is optional; agent-to-agent review handles most validation

This creates a "[[Ralph Wigum Loop]]"—an iterative feedback cycle where agents validate their own work before human intervention.

## Making Applications Legible to Agents

### The Legibility Problem

Early development revealed that progress was limited not by Codex's capabilities, but by environmental underspecification. Agents lacked the tools, abstractions, and information structure needed to accomplish complex tasks.

The solution: **increase application legibility**—make the system directly observable and manipulable by agents.

### UI and Observability Integration

The team wired agents directly into the application's operational characteristics:

- **Chrome DevTools Protocol integration**: Agents can launch app instances per [[git worktree]], drive UI interactions, capture screenshots, and reason about DOM snapshots
- **Observability stack access**: Logs, metrics, and traces exposed to agents via local [[LogQL]] and [[PromQL]] queries
- **Isolated testing environments**: Each agent task runs on a fully isolated application instance with its own logs and metrics, torn down after completion
- **Extended execution**: Single Codex runs regularly work on tasks for 6+ hours (often while humans sleep)

This enabled agents to tackle prompts like "ensure service startup completes under 800ms" or "no span in critical user journeys exceeds two seconds"—directly validating fixes against observable behavior.

## Repository as System of Record

### The AGENTS.md Problem

The team initially attempted to provide comprehensive guidance through a single `AGENTS.md` instruction file. This approach failed predictably:

- **Context crowding**: Large instruction files displaced the actual task and code from the agent's working context
- **Signal loss**: When everything is marked important, nothing is—agents pattern-matched locally rather than navigating intentionally
- **Rapid rot**: Monolithic manuals became graveyards of stale rules that agents couldn't distinguish from current guidance
- **Verification challenges**: Single-blob documentation resisted mechanical validation and drift detection

### Progressive Disclosure Model

Instead, the team treated `AGENTS.md` as a **table of contents** (~100 lines) and distributed knowledge throughout a structured `docs/` directory:

```
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   └── uv-llms.txt
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
└── QUALITY_SCORE.md
```

### Knowledge Organization Principles

- **Design documentation**: Catalogued and indexed with verification status and core agent-first operating principles
- **Architecture documentation**: Top-level mapping of domains and package layering
- **Quality tracking**: Domain and layer grades tracking gaps over time
- **Execution plans**: First-class artifacts with progress and decision logs, versioned and co-located
- **Mechanical enforcement**: Dedicated linters and CI jobs validate that documentation is up-to-date, cross-linked, and correctly structured
- **Doc gardening**: Recurring agent tasks scan for stale documentation and open fix-up pull requests

This approach enables **progressive disclosure**: agents start with a small, stable entry point and learn where to look next, rather than being overwhelmed upfront.

## Agent Legibility as Design Goal

### Repository Optimization for Agents

Because the entire codebase was agent-generated, it was optimized first for [[Codex]]'s legibility rather than human preferences. The guiding principle: *anything the agent can't access in-context while running effectively doesn't exist*.

Knowledge living in external systems—Google Docs, chat threads, people's heads—is inaccessible and therefore illegible to agents. Only repository-local, versioned artifacts (code, markdown, schemas, executable plans) are visible.

### Technology Selection for Legibility

This framing clarified technology choices:

- **Preference for "boring" tech**: Technologies with composable APIs, stable interfaces, and strong training set representation are easier for agents to model
- **Reimplementation over opacity**: Rather than using generic third-party packages with opaque behavior, the team sometimes had agents reimplement functionality tailored to the system's needs
  - Example: Custom `map-with-concurrency` helper instead of generic `p-limit` package, with 100% test coverage and tight [[OpenTelemetry]] integration
- **Internalization over external dependency**: Dependencies must be fully internalized and reasoned about within the repository

## Enforcing Architecture and Taste

### Rigid Architectural Models

Rather than relying on documentation alone to maintain coherence in an agent-generated codebase, the team built enforcement directly into the system. Agents are most effective in environments with **strict boundaries and predictable structure**.

### Layered Architecture Pattern

Each business domain is divided into a fixed set of layers with strictly validated dependency directions:

```
Types → Config → Repo → Service → Runtime → UI
```

Rules:
- Code can only depend "forward" through this fixed sequence
- Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through a single explicit interface: **Providers**
- All other dependencies are disallowed and mechanically enforced

### Mechanical Enforcement

Constraints are enforced through:

- **Custom linters**: Generated by Codex itself, with error messages that inject remediation instructions into agent context
- **Structural tests**: Verify architectural invariants
- **Taste invariants**: Statically enforced rules including:
  - Structured logging requirements
  - Naming conventions for schemas and types
  - File size limits
  - Platform-specific reliability requirements

### Enforcement Philosophy

The team is explicit about where constraints matter and where they don't—resembling a large engineering platform organization:

- **Enforce boundaries centrally**: Architectural constraints, correctness, reproducibility
- **Allow autonomy
