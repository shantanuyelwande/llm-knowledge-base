# Harness Engineering: Leveraging Codex in an Agent-First World

## Summary

**Harness engineering** is an approach to software development where [[AI agents]] (specifically [[Codex]]) generate and maintain code rather than human engineers writing it manually. This methodology shifts the primary role of engineers from code production to designing systems, specifying intent, and building feedback loops that enable agents to work reliably. An internal team at [[OpenAI]] successfully built a production software product with zero manually-written lines of code over five months, generating approximately one million lines across all aspects of development.

---

## Overview

Harness engineering represents a fundamental shift in how software teams operate when [[AI code generation]] becomes the primary development mechanism. Rather than humans writing code directly, engineers act as architects and stewards of the development environment, enabling AI agents to produce reliable, maintainable software at scale.

### Core Philosophy

The defining principle of harness engineering is **no manually-written code**. Every line of code—from application logic to tests, CI configuration, documentation, and tooling—is generated by agents. This constraint forces teams to invest in the infrastructure, abstractions, and feedback systems necessary for agents to work effectively.

## Key Principles and Practices

### 1. Redefining the Engineer's Role

In an agent-first development model, human engineers transition from direct code production to **systems engineering**:

- **Prompt-based interaction**: Engineers describe tasks in natural language and allow agents to implement them
- **Environment design**: Creating tools, abstractions, and structures that make agent work possible
- **Feedback loop management**: Establishing mechanisms for agents to validate, review, and iterate on their work
- **Problem diagnosis**: When agents fail, engineers identify missing capabilities rather than attempting workarounds

The team found that early progress was slower than expected not due to agent limitations, but because the development environment was underspecified. Success came from systematically building the tools and context agents needed.

### 2. Making Applications Legible to Agents

A critical challenge in agent-first engineering is ensuring agents can understand and reason about the system. This requires making normally human-facing elements directly accessible to agents:

**Application UI Legibility**
- Applications are bootable per [[git]] worktree, allowing agents to launch isolated instances
- [[Chrome DevTools Protocol]] integration enables agents to work with DOM snapshots and screenshots
- Agents can reproduce bugs, validate fixes, and reason about UI behavior directly

**Observability Integration**
- Logs, metrics, and traces are exposed via local observability stacks
- Agents query logs using [[LogQL]] and metrics using [[PromQL]]
- Performance constraints become tractable: "ensure service startup completes in under 800ms"
- Single Codex runs regularly extend beyond six hours for complex tasks

### 3. Repository as System of Record

[[Knowledge management]] is crucial for agent effectiveness. Rather than maintaining monolithic instruction files, the team implemented a structured approach:

**Problems with Centralized Instructions**
- Context is scarce; large files crowd out relevant code and documentation
- Excessive guidance becomes non-guidance when everything is marked "important"
- Maintenance decay: instruction files become graveyards of stale rules
- Difficult to verify completeness and accuracy mechanically

**Structured Documentation Approach**
- `AGENTS.md` (~100 lines) serves as a table of contents and entry point
- `docs/` directory is the system of record, structured as:
  - `design-docs/`: Indexed design documentation with verification status
  - `exec-plans/`: First-class artifacts for complex work with decision logs
  - `product-specs/`: Product requirements and specifications
  - `references/`: Technology-specific reference materials
  - Generated documentation (e.g., database schemas)

**Mechanical Enforcement**
- Custom [[linters]] validate documentation is up-to-date and cross-linked
- CI jobs ensure structural correctness
- Recurring "doc-gardening" agents scan for stale documentation and open fix-up pull requests

### 4. Agent Legibility as Design Constraint

The entire codebase is optimized for [[agent readability]] rather than human stylistic preferences. This shapes architectural and technology choices:

**Design Principles**
- Anything not accessible in-context while running effectively doesn't exist for the agent
- Knowledge in external systems (Google Docs, Slack, people's heads) is illegible
- Repository-local, versioned artifacts are the only reliable knowledge source
- Technologies described as "boring" are preferred due to composability and API stability

**Technology Decisions**
- Favor dependencies that can be fully internalized and reasoned about in-repository
- Sometimes cheaper to reimplement functionality than work around opaque upstream behavior
- Example: Custom `map-with-concurrency` helper instead of generic `p-limit` package, with 100% test coverage and tight OpenTelemetry integration

### 5. Enforcing Architecture and Taste

Maintaining coherence in a fully agent-generated codebase requires explicit architectural constraints and mechanical enforcement:

**Rigid Architectural Model**
- Each business domain divided into fixed layers: Types → Config → Repo → Service → Runtime → UI
- Strictly validated dependency directions
- Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through single explicit interface: Providers
- All other dependencies disallowed and mechanically enforced

**Taste Invariants**
- Custom [[linters]] enforce structured logging, naming conventions, file size limits
- Platform-specific reliability requirements encoded as lints
- Error messages include remediation instructions for agents
- Rules apply everywhere simultaneously once encoded

**Balancing Constraints and Autonomy**
- Enforce boundaries centrally, allow autonomy locally
- Care deeply about boundaries, correctness, and reproducibility
- Within boundaries, allow agents significant freedom in implementation
- Code may not match human stylistic preferences, but must be correct, maintainable, and legible

### 6. Merge Philosophy for High-Throughput Environments

Traditional engineering norms become counterproductive when agent throughput far exceeds human attention:

**Minimal Blocking Gates**
- Pull requests are short-lived
- Test flakes addressed with follow-up runs rather than blocking indefinitely
- Corrections are cheap; waiting is expensive
- This tradeoff is responsible because high throughput enables rapid remediation

**Throughput Metrics**
- ~1,500 pull requests merged over five months
- 3.5 PRs per engineer per day average
- Throughput increased as team grew from 3 to 7 engineers

### 7. Escalating Agent Autonomy

As development infrastructure matured, agents achieved end-to-end autonomy for feature delivery:

**Full Autonomy Workflow**
Given a single prompt, agents can:
- Validate current codebase state
- Reproduce reported bugs
- Record demonstration videos
- Implement fixes
- Validate fixes by driving the application
- Open pull requests with supporting evidence
- Respond to agent and human feedback
- Detect and remediate build failures
- Merge changes when appropriate
- Escalate to humans only when judgment is required

**Important Caveat**: This level of autonomy depends heavily on specific repository structure and tooling investment. Generalization requires similar infrastructure investment.

### 8. Managing Entropy and Technical Debt

Full agent autonomy introduces novel challenges around code quality and consistency:

**The Problem**
- Agents replicate patterns existing in the repository, even suboptimal ones
- Patterns drift over time as agents make local decisions
- Manual cleanup (20% of time each Friday) proved unscalable

**Garbage Collection Approach**
- "Golden principles": Opinionated, mechanical rules encoded directly in the repository
- Examples: Prefer shared utility packages over hand-rolled helpers; validate data boundaries rather than probing "YOLO-style"
- Recurring background Codex tasks scan for deviations, update quality grades, open targeted refactoring PRs
- Technical debt paid down continuously in small increments rather than accumulating
- Human taste captured once, enforced continuously on every line of code

## What Gets Generated

The claim of "zero manually-written code" encompasses:

- Product code and tests
- [[CI/CD]] configuration and release tooling
- Internal developer tools
- Documentation and design history
- Evaluation harnesses
- Review comments and responses
- Repository management scripts
- Production dashboard definitions

Humans remain in the loop but work at a different abstraction layer:
- Prioritize work
- Translate user feedback into acceptance criteria
- Validate outcomes
- Identify missing capabilities when agents struggle

## Key Learnings and Open Questions

### What Worked Well

- **Structured documentation** dramatically improved agent effectiveness
- **Mechanical enforcement** of architecture and taste scaled better than human code review
- **Progressive disclosure** of information (map → detailed docs) outperformed monolithic instructions
- **Isolated environments** (per-worktree instances) enabled agents to validate their own work
- **High throughput** made rapid iteration and correction feasible
- **Real product use**
