# Building with Modal and the OpenAI Agents SDK

## Summary

This article explores how to build custom [[AI agent]] harnesses using the [[OpenAI Agents SDK]] integrated with [[Modal]] for scalable, parallel execution. It demonstrates a progression from basic coding agents to sophisticated multi-agent systems with memory management, GPU acceleration, and filesystem snapshots for efficient autonomous research tasks.

---

## Overview

The [[OpenAI Agents SDK]] provides powerful building blocks for creating custom agentic systems. When combined with [[Modal]]'s [[Sandbox|Sandboxes]] and [[GPU]] infrastructure, developers can build production-grade agent harnesses capable of handling complex tasks like the OpenAI [[Parameter Golf]] challenge—optimizing neural networks to achieve intelligence thresholds with minimal parameters.

This guide demonstrates how companies like [[Ramp]] build internal coding agents that automate significant portions of their development workflows, with over half of pull requests generated by autonomous agents.

---

## Core Concepts

### What is an Agent Harness?

An **agent harness** is the complete system built around a core [[agent]] loop. It consists of:

- The base [[agent]] (a loop running an [[LLM]] with tools)
- Tools (functions the agent can invoke)
- State management
- Context handling
- Security boundaries

The harness is where product engineering happens—it's entirely under programmer control and can be customized for specific tasks.

## Building Progression

### 1. Basic Coding Agent

The simplest coding agent is a loop where an [[LLM]] can invoke an `exec()` function to run arbitrary shell commands:

```python
agent = Agent(tools=[exec_tool])
agent.run("train a model on MNIST")
```

**Note:** This approach is unsafe and not recommended for production without proper safeguards.

### 2. Sandboxed Execution

[[Modal]] [[Sandbox|Sandboxes]] are isolated Linux environments built on VMs or security-hardened containers. Moving agent execution into sandboxes provides:

- **Security isolation**: The [[LLM]] operates within a contained environment rather than on the host system
- **Resource control**: Ability to attach [[GPU|GPUs]] and manage computational resources
- **Stateful tools**: The OpenAI Agents SDK provides `SandboxAgent` class with `ShellTool` for safer command execution

#### GPU-Enabled Sandboxes

[[Modal]] allows attaching [[GPU|GPUs]] to sandboxes using `ModalSandboxClientOptions`, enabling agents to perform computationally intensive tasks like model training directly.

### 3. Memory and Sessions

By default, [[agent|agents]] are stateless. The `Session` object solves this by:

- Accumulating conversation context across multiple runs
- Maintaining state across user prompts
- Enabling multi-turn interactions

**Challenge introduced:** [[Context rot]] occurs as memory accumulates indefinitely, requiring intelligent context management strategies.

### 4. Orchestrator and Subagent Architecture

To handle long-horizon tasks and manage [[token]] consumption:

- **Orchestrator agent**: Main [[agent]] maintaining high-level task memory
- **Subagents**: Spawned for specific focused tasks with fresh context windows
- **Delegation tool**: `invoke_subagent` allows the orchestrator to delegate work

**Benefits:**
- Prevents context bloat in the main agent
- Allows subagent sessions to be discarded after task completion
- Keeps orchestrator focused on high-level planning

### 5. Async Subagent Pool

Rather than blocking the orchestrator while subagents work:

- Implement a `SubAgentPool` managing multiple parallel subagents
- Use `asyncio.Future` objects for non-blocking task execution
- Orchestrator can spawn multiple subagents and monitor their progress

**Visibility tools:**
- `list_subagents`: Shows active subagents and their status
- `set_status`: Allows subagents to update progress without exiting
- [[Hook|Hooks]]: Track current tool execution for each subagent

### 6. GPU Quota Management

With async parallel subagents, implement quota systems to:

- Limit concurrent expensive [[GPU]] usage
- Prevent unbounded resource consumption
- Control costs while maintaining parallelism

### 7. Filesystem Snapshots

[[Filesystem snapshots]] freeze the state of an active [[Sandbox]] into an ID, enabling:

- **Work deduplication**: Avoid repeated setup (repo pulls, dependency installation) across subagents
- **Checkpointing**: Branch work from known stable states
- **Implicit memory**: Filesystem acts as persistent state accessible to future agents
- **Context offloading**: Reduce [[context window]] pressure by storing artifacts on disk

Subagents can be spawned from snapshots with minimal instructions and quickly resume work from known states.

### 8. Skills Subsystem

A pluggable system for adding task-specific knowledge:

- Keeps the core harness general-purpose
- Allows orchestrator to selectively opt-in to domain-specific context
- Enables reusable skill plugins for different problem domains

## Key Design Patterns

### Context Management Strategy

- Keep orchestrator context lean and high-level
- Delegate detailed work to short-lived subagents
- Use filesystem snapshots for state persistence
- Implement quota systems to prevent resource overload

### Parallelization Approach

- Async subagent pools for non-blocking execution
- Status tracking tools for visibility
- Shared snapshots as starting points for related work
- GPU quotas to manage costs

### State Handling

- **Session-based**: For conversational context
- **Filesystem-based**: For artifacts and implicit memory
- **Ephemeral**: Subagent sessions discarded after use
