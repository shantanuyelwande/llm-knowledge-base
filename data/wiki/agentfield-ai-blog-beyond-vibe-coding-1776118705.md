# Multi-Agent Code Generation and Orchestration

## Summary

This article explores the architectural patterns and lessons learned from orchestrating 200+ autonomous [[AI agent|AI agents]] to collaboratively write production code. The work, conducted by the [[AgentField]] team, demonstrates that successful multi-agent systems require two distinct modes of [[LLM integration]], nested failure recovery loops, and checkpoint-based execution for cost-effective, durable workflows.

---

## Overview

As [[AI-native engineering]] teams scale autonomous code generation from single-agent to multi-agent systems, new challenges emerge around convergence, failure recovery, and cost management. The convergence problem—getting N autonomous processes to produce one coherent result—requires specialized primitives for isolation, failure recovery, and state reconciliation.

This work emerged from building [[SWE-AF]], an open-source system that ran real production builds including a Rust diagrams-as-code CLI (15 issues, 200+ agent invocations, $116), a Go SDK feature (10 issues, 80+ invocations, $19), and a Node.js benchmark (95/100 score on both cheapest and mid-tier models).

---

## Core Architectural Patterns

### Two Modes of LLM Integration

The most critical architectural decision is separating [[LLM integration]] into two distinct primitives rather than giving all agents uniform access.

#### Constrained Single-Shot Calls (`.ai()`)

The first primitive is a **constrained call**: single-shot, structured input/output, no tools, no iteration. These calls handle routing and classification tasks with:

- **Predictable latency**: millisecond response times
- **Predictable cost**: fractions of a cent per call
- **Deterministic outputs**: structured fields that downstream code can switch on

Example guidance block generated by a constrained call:

```
IssueGuidance:
  needs_new_tests: true
  estimated_scope: "medium"
  touches_interfaces: true
  needs_deeper_qa: true
  agent_guidance: "Complex parsing logic with edge cases, run QA in parallel"
```

These calls route work efficiently and carry context for downstream agents without expensive iteration.

#### Autonomous Harnesses (`.harness()`)

The second primitive is an **autonomous loop** (harness): multi-turn, tool-using, goal-driven. A harness receives a goal and toolset, then iterates until producing a verifiable outcome. It:

- Reads and writes files
- Runs tests and discovers failures
- Attempts repairs and retries
- Produces verifiable deliverables

Key characteristics:

- **Variable cost**: single harness invocation can run 50-150 tool-use turns
- **Variable duration**: from minutes to hours depending on complexity
- **Outcome-focused**: the system checks what was delivered, not how it was achieved

In the diagrams build, a single coder harness invocation cost over $4 on complex issues, running up to 150 tool-use turns.

#### Routing and Flexibility

These primitives coexist in the same system. A cheap `.ai()` classification call determines issue routing:

- **Lean path** (2 calls): coder → reviewer
- **Thorough path** (4 calls): coder → QA and reviewer (parallel) → synthesizer

The routing cost is negligible; the expensive work is conditional and proportional to risk.

## Failure Recovery and Escalation

In a 200+ invocation build, failures are the normal path, not edge cases. The system requires three nested control loops to handle different failure classes.

### The Inner Loop: Per-Issue Iteration

The inner loop runs up to 5 iterations per issue. The agent receives feedback from [[QA verification]] and code review, then retries with improved information.

**Example**: The app-module issue passed all 119 tests and met acceptance criteria, but the code reviewer caught that the module was not exported in `lib.rs`. The inner loop caught this real, fixable problem and iteration 2 succeeded with 354 tests passing.

### The Middle Loop: Issue Advisor

When the inner loop exhausts iterations, an **issue advisor** activates with five typed recovery actions:

| Action | Meaning |
|--------|---------|
| `RETRY_MODIFIED` | Relax acceptance criteria; record gap as typed debt |
| `RETRY_APPROACH` | Keep criteria; try different strategy or library |
| `SPLIT` | Break issue into smaller sub-issues |
| `ACCEPT_WITH_DEBT` | Close enough; record each gap as typed, severity-rated debt |
| `ESCALATE_TO_REPLAN` | Cannot fix locally; restructure remaining work |

The advisor's final invocation explicitly warns it is the last chance, biasing toward acceptance or escalation rather than futile retry.

### The Outer Loop: Replanning

The **replanner** fires when issues in a dependency level produce unrecoverable failures. It has full execution state visibility and can:

- Skip downstream dependents
- Restructure the remaining issue graph
- Reduce scope
- Abort if necessary

Previous replan decisions are fed back on subsequent invocations to prevent repeating failed strategies.

### Failure Case Studies

#### Deadlocks That Retrying Cannot Fix

The integration-tests issue timed out after 2700 seconds. The tests were correct and code-review approved, but the CLI binary had an infinite loop. Retrying the same agent on the same binary produced the same timeout.

**Response**: The system **blocked** the issue, recorded typed debt, and let the rest of the build work around the gap:

```json
{
  "type": "dropped_acceptance_criterion",
  "criterion": "integration tests pass end-to-end",
  "issue_name": "integration-tests",
  "severity": "high",
  "justification": "CLI binary deadlocks on invocation; test suite correct but binary needs runtime debugging beyond automated repair"
}
```

#### Review Catches That Tests Miss

The app-module issue had 119 passing tests and met all acceptance criteria, but was not exported in `lib.rs`. The tests passed because they mocked the dependency. The code reviewer caught the structural problem, and iteration 2 fixed it.

#### Regressions on Trivial Tasks

The project-scaffold issue (simply "create Cargo.toml and src/main.rs") passed iteration 1 with all 13 tests passing. Iteration 2 regressed by adding module declarations for non-existent code, causing cargo build to fail.

#### Cascading Verification Failures

The final-acceptance-verification issue required three iterations:
1. Fixed `set -e` crashes, macOS-specific `stat` flags, infinite recursion, missing dependency checks
2. Introduced new bug with `cargo fmt` using `|| true` (always passing)
3. Finally got it right

## Durable Execution and Cost Management

At $116 per build, restarting from scratch after a failure at invocation 140 is not viable. The system treats multi-agent builds as long-running, expensive processes requiring checkpoint-based recovery.

### Checkpoint Everything

The system checkpoints at every dependency level boundary, capturing:

```json
{
  "current_level": 3,
  "completed_levels": [0, 1, 2],
  "all_issues": ["project-scaffold", "types-module", ...],
  "original_plan_summary": "15 issues organized in 6 levels...",
  "replan_count": 0,
  "accumulated_debt": [...]
}
```

`resume_build()` loads the checkpoint, skips completed levels, and continues from the exact failure point. A 30-minute build failing at minute 25 does not restart from minute 0.

The checkpoint also captures [[git]] state (integration branch, original branch, initial commit SHA, worktree directory mapping) for full workspace reconstruction.

### Isolation with Git Worktrees

Each issue gets its own [[git worktree]] on a dedicated branch (`issue/01-project-scaffold`, `issue/02-types-module`). This prevents lock contention and conflicts during parallel coding.

**Planning insight**: Agents consistently produce finer-grained, more parallelizable issue graphs than humans would attempt. The planner decomposes epics into 50-100 issues with dependency structures that humans would avoid due to coordination overhead. The orchestrator absorbs this coordination cost through scheduling.
