# Multi-Agent Code Generation and Orchestration

## Summary

This article documents how [[AgentField]] orchestrates 200+ autonomous [[LLM]] agents to collaboratively produce production-quality code. The key innovation separates [[LLM]] integration into two primitives: constrained single-shot calls (`.ai()`) for routing and classification, and autonomous harnesses (`.harness()`) for iterative code generation. The system uses three nested failure loops, checkpoint-based recovery, and git worktrees for isolation to handle the complexity of coordinating multiple agents on shared codebases.

---

## Overview

[[AgentField]] is an AI-native engineering platform that abstracts human developers upward from the iteration loop to the review loop. Rather than having individual engineers guide [[LLM]] sessions through code changes, the platform orchestrates dozens of autonomous agents working in parallel on a shared codebase. These agents, called **harnesses**, are full coding environments with filesystem access, test execution capabilities, and git integration.

The fundamental challenge is the **convergence problem**: getting N autonomous processes to produce one coherent result requires explicit primitives for isolation, failure recovery, and state reconciliation.

### Key Achievement

In production deployments, the system has successfully:
- Processed a Rust diagrams-as-code CLI with 15 issues across 6 dependency levels using 200+ agent invocations at a cost of $116
- Completed a Go SDK feature with 10 issues using 80+ invocations for $19
- Achieved 95/100 on a Node.js benchmark using both the cheapest and mid-tier LLM models

## Two Modes of LLM Integration

The most critical architectural decision is separating [[LLM]] access into two distinct primitives, each with different characteristics and use cases.

### The Constrained Call: `.ai()`

A **constrained call** is a single-shot, deterministic interaction with the [[LLM]]:

- **Input**: Structured, bounded
- **Output**: Structured, predictable schema
- **Tools**: None
- **Iteration**: Single turn only
- **Cost**: Fractions of a cent
- **Latency**: Milliseconds
- **Use case**: Routing, classification, decision-making

Example guidance block generated by a constrained call:

```yaml
IssueGuidance:
  needs_new_tests: true
  estimated_scope: "medium"
  touches_interfaces: true
  needs_deeper_qa: true
  agent_guidance: "Complex parsing logic with edge cases, run QA in parallel"
```

The structured output fields drive routing decisions. The `agent_guidance` string carries semantic context to downstream agents. This primitive enables operational predictability: you can set [[SLAs]], predict costs, and build reliable retry logic.

### The Autonomous Loop: `.harness()`

An **autonomous harness** is a multi-turn, tool-using agent with a full coding environment:

- **Input**: Goal statement and toolset
- **Output**: Verifiable outcome (passing tests, compiled code)
- **Tools**: File system access, test execution, git operations, code review
- **Iteration**: Multi-turn, self-directed until goal achieved
- **Cost**: Dollars per invocation (typically $0.50–$4.26 per issue)
- **Latency**: Minutes to hours
- **Use case**: Code generation, testing, implementation

The harness receives a goal and iterates until producing a verifiable outcome. Success is measured by external validation (tests pass, code compiles, review approves), not by the number of turns taken.

### Why Both Matter

The constrained call is not novel—every [[agentic framework]] has some version. The harness is the critical innovation. It emerged from production failures rather than design exercises. The two coexist in the same system: a cheap boolean from `.ai()` determines whether an issue follows a lean path (coder → reviewer) or a thorough path (coder → QA + reviewer in parallel → synthesizer).

## Failure Handling and Escalation

In a 200+ invocation build, failures are the normal path, not edge cases. The system requires three nested control loops to handle the diversity of failure modes.

### Failure Types in Practice

#### Deadlocks That Retrying Cannot Fix

When an agent produces code with an infinite loop or deadlock, retrying the same agent produces the same failure. The integration-tests issue in the diagrams build exemplified this: tests timed out after 2700 seconds, code review approved the test suite, but the underlying CLI binary hung.

**Response**: Block the issue, record typed debt, and continue the build.

#### Review Catches That Tests Miss

The app-module issue passed all 119 tests and met acceptance criteria, but the code reviewer discovered the module was not exported in `lib.rs`. Every test passed because tests mocked the dependency. The module was invisible to library consumers.

**Response**: Inner retry loop fixes the issue on the next iteration.

#### Regressions on Trivial Tasks

The project-scaffold issue ("create Cargo.toml and src/main.rs") passed iteration 1 but regressed in iteration 2 when the agent added module declarations for code that did not exist yet.

**Response**: Inner retry loop catches and fixes the regression.

#### Cascading Verification Failures

The final-acceptance-verification issue required three iterations: iteration 1 found four blocking issues, iteration 2 fixed those but introduced `|| true` making checks always pass, iteration 3 finally succeeded.

**Response**: Escalation through middle loop to advisor agent.

### Three Nested Control Loops

```
┌─────────────────────────────────────────────────┐
│ Outer Loop: Replanner (Full Build State)        │
│ Triggers on: Unrecoverable failures at level    │
│ Actions: Skip dependents, restructure graph,    │
│          reduce scope, abort if necessary       │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│ Middle Loop: Issue Advisor (Exhausted Retries)  │
│ Triggers on: Inner loop exhausted (5 attempts)  │
│ Actions: RETRY_MODIFIED, RETRY_APPROACH,        │
│          SPLIT, ACCEPT_WITH_DEBT, ESCALATE      │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│ Inner Loop: Agent Self-Retry (Per Issue)        │
│ Triggers on: QA failure or review blocking      │
│ Actions: Iterate with feedback, up to 5 times   │
└─────────────────────────────────────────────────┘
```

#### Inner Loop: Per-Issue Iteration

Runs up to 5 iterations per issue. The agent receives feedback from [[QA]] and code review, then attempts to fix the identified problems. Handles issues the same agent can solve with better information.

#### Middle Loop: Issue Advisor

Activates when the inner loop exhausts all retries. Has five typed recovery actions:

| Action | Behavior |
|--------|----------|
| `RETRY_MODIFIED` | Relax acceptance criteria, record gap as typed debt |
| `RETRY_APPROACH` | Same criteria, different strategy (e.g., different library) |
| `SPLIT` | Break issue into smaller sub-issues |
| `ACCEPT_WITH_DEBT` | Close enough; record each gap as typed, severity-rated debt |
| `ESCALATE_TO_REPLAN` | Cannot fix locally; restructure remaining work |

The final invocation explicitly warns this is the last chance, biasing toward acceptance or escalation rather than futile retries.

#### Outer Loop: Replanner

Fires when issues in a dependency level produce unrecoverable failures. Sees full execution state and can skip downstream dependents, restructure the issue graph, reduce scope, or abort. Previous replan decisions are fed back on subsequent invocations to prevent repeating failed strategies.

If the replanner itself crashes (LLM timeout, malformed output), the system defaults to **continue** rather than abort. For expensive workflows, graceful degradation is better than fail-fast.

### Typed Debt Items

Instead of silently dropping failures or aborting builds, the system records structured debt:

```json
{
  "type": "dropped_acceptance_criterion",
  "criterion": "integration tests pass end-to-end",
  "issue_name": "integration-tests",
  "severity": "high",
  "justification": "CLI binary deadlocks on invocation; test suite correct but binary needs runtime debugging beyond automated repair"
}
```
